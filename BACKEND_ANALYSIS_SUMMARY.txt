## API Endpoints Summary

### Core Endpoints

1. **POST /api/v1/step/upload**
   - Upload STEP CAD file (asynchronous)
   - Returns: job_id, model_id, status, estimated_time_seconds
   - Validation: File size (max 50 MB), STEP format (ISO-10303-21)
   - Triggers background 4-stage pipeline

2. **GET /api/v1/step/progress/{job_id}/stream**
   - Real-time progress via Server-Sent Events (SSE)
   - Returns: Streaming JSON events with progress_percent, current_stage, action, eta_seconds
   - Stages: loading_geometry (0-25%), extracting_parts (25-50%), generating_svgs (50-75%), generating_assembly (75-100%)
   - Status values: processing, complete, failed

3. **GET /api/v1/jobs/{job_id}**
   - Single snapshot of job status
   - Returns: job_id, model_id, status, progress_percent, current_stage, action, eta_seconds, error_message
   - Non-streaming alternative to SSE endpoint

4. **GET /api/v1/step/{model_id}**
   - Retrieve processed geometry data
   - Returns: Geometry3D with vertices, normals, indices, metadata
   - Geometry: Triangle mesh representation of STEP model
   - Metadata: Contains solids with bounding boxes, centroids, volume, surface area

5. **POST /api/v1/step/parts-2d**
   - Extract and classify parts from model
   - Request: {model_id} or {modelId}
   - Returns: PartsResponse with parts list (id, part_type, quantity, volume, dimensions)
   - Part types: panel, hardware, fastener, structural, other
   - Includes deduplication with adaptive tolerance (5-30%)

6. **POST /api/v1/step/assembly-analysis**
   - Generate assembly instructions
   - Request: {model_id}, optional preview_only parameter
   - Returns: AssemblyResponse with assembly steps
   - Phase 3 LLM features: detail_description, assembly_sequence, warnings, tips, confidence_score, is_llm_generated
   - Modes: Rules-based (default) or LLM-powered (if enabled)

7. **GET /api/v1/health**
   - Health check endpoint
   - Returns: {status: "healthy", version: "0.1.0", message}

8. **GET /api/v1/health/detailed**
   - Detailed health check with service status
   - Returns: {status, services: {api, database, storage}}

## Processing Pipeline

### Stage 1: STEP Loading (0-25%)
- Input: Raw STEP file bytes
- Process: CadQuery library or fallback text parser
- Output: Geometry3D {vertices, normals, indices, metadata}
- Metadata: Contains solids with bounding boxes, centroids, volume, surface area

### Stage 2: Parts Extraction (25-50%)
- Input: Geometry3D with solids metadata
- Process: Classification and deduplication
- Output: Parts[] {id, part_type, quantity, volume, dimensions, centroid, surface_area, group_id}
- Classification: Panel, Hardware, Fastener, Structural, Other
- Deduplication: Adaptive tolerance based on model scale

### Stage 3: SVG Generation (50-75%)
- Input: Parts[] with dimensions
- Process: Generate technical drawings
- Output: SvgDrawing[] {part_id, svg_content, quantity_label, metadata}
- Drawings: Orthographic projections + isometric view with dimension annotations

### Stage 4: Assembly Generation (75-100%)
- Input: Parts[] and SvgDrawing[]
- Process: Rules-based or LLM-powered sequence generation
- Output: AssemblyStep[] {step_number, title, description, part_indices, part_roles, context_part_indices, duration_minutes, ...}
- LLM additions: detail_description, assembly_sequence, warnings, tips, confidence_score

## Field Name Inconsistencies

1. **model_id vs modelId**
   - Backend stores internally as: model_id
   - Accepts both in request: model_id or modelId
   - Endpoints: POST parts-2d, POST assembly-analysis
   - Resolution: payload.get("modelId") or payload.get("model_id")

2. **job_id vs jobId**
   - Consistently uses: job_id
   - Path parameter: {job_id}

3. **progress_percent vs percentage**
   - Internal (ProgressEvent): percentage
   - API (ProgressStreamResponse): progress_percent
   - Frontend expects: percentage
   - Transformation: In stream_progress endpoint

4. **current_stage vs stage**
   - Internal (ProgressEvent): stage
   - API (ProgressStreamResponse): current_stage
   - Frontend display: stage name

5. **action vs message**
   - Internal (ProgressEvent): message
   - API (ProgressStreamResponse): action
   - Frontend display: action text

## Response Format Differences

### Geometry Response Format
GET /api/v1/step/{model_id} wraps geometry in outer object:
`json
{
  "status": "success",
  "model_id": "...",
  "file_name": "...",
  "file_size": 123456,
  "geometry_loaded": true,
  "geometry": {
    "vertices": [...],
    "normals": [...],
    "indices": [...],
    "metadata": {...}
  }
}
`

Internally stored as Geometry3D (flat):
`python
@dataclass
class Geometry3D:
    vertices: List[List[float]]
    normals: List[List[float]]
    indices: List[int]
    metadata: Dict[str, Any]
`

### Status Values Mismatch
- SSE final event: status="success"
- JobStatusResponse: status="complete"
- ProcessingStatus enum: "complete"
- Typically handled as: "success" mapped to "complete" in API

## Critical Findings

### Data Consistency Issues

1. **Field naming is inconsistent**
   - Some endpoints accept both snake_case and camelCase
   - Creates confusion for API consumers
   - Mitigation: Backend normalizes, but not documented

2. **SSE response differs from job status response**
   - Same data but different field names
   - Transformation happens in endpoint
   - Not obvious from API documentation

3. **Geometry wrapping is asymmetric**
   - Response wraps Geometry3D in outer object
   - Internal storage is flat
   - Manual wrapping in endpoint adds complexity

4. **Status values use different terminology**
   - "success" vs "complete"
   - Could cause client-side bugs if not handled
   - Should standardize on one term

### LLM Integration Notes

- Phase 3 LLM features are optional (assembly_llm_enabled config)
- Falls back to rules-based if LLM fails
- Confidence scores differentiate LLM (0.85-0.95) vs rules (0.8)
- is_llm_generated flag allows frontend to show quality indicators
- Tone settings affect instruction style (IKEA vs TECHNICAL vs BEGINNER)

### Performance Considerations

- STEP loading timeout: 5 minutes (handles large files)
- Assembly generation timeout: 2 minutes (LLM calls are slower)
- SSE queue size: 100 events max (bounded memory)
- Dead queue removal prevents subscriber leaks

## Recommendations

1. **Standardize field names**
   - Use snake_case consistently
   - Document camelCase aliases if needed
   - Avoid mixing in same response

2. **Document transformations**
   - Clearly note where field names change
   - Create mapping table in API docs
   - Use consistent terminology (status values)

3. **Add OpenAPI/Swagger spec**
   - Auto-generate from Pydantic models
   - Provides single source of truth
   - Easier for frontend teams to consume

4. **Version error responses**
   - Current error format may change
   - Add error code documentation
   - Version error schema separately

5. **Add request ID tracking**
   - Would help with debugging
   - Could trace errors through pipeline
   - Useful for metrics/analytics
